# PromptPro TODO

## 现状判断

这个项目当前最大的问题不是“代码风格不够漂亮”，而是 **产品定位过大、实现边界不清、代码结构和运行副作用耦合得太紧**。

从现状看，它试图同时做这些事：

- Prompt 优化器
- Prompt 框架推荐器
- 多 Provider LLM CLI
- 历史记录工具
- 需求文档管理器
- 交互式问答助手

但实际实现还是偏单体脚本式 CLI，导致几个后果：

- README 的产品承诺明显大于代码真实能力
- 核心逻辑和 UI、配置、文件系统、网络请求强耦合
- 模块导入就触发真实副作用，测试和复用都很差
- “智能推荐 / 智能优化”本质还是硬编码规则 + 远程调用包装，产品差异化不成立

结论：**先收缩产品，再拆结构，再补质量。**

## 为什么现在会越改越乱

这个项目已经出现了比较典型的“小工具失控”迹象：

- 一开始是 CLI 小工具，后面不断往里叠功能
- 每个功能都能单独解释，但没有统一的内核模型
- 为了让功能尽快可用，很多逻辑直接写进 CLI 或模块全局
- 结果是新增功能很快，稳定性和可维护性却越来越差

这类项目最常见的恶化路径是：

1. 先在 `cli.py` 里写一个功能分支。
2. 为了省事，直接 import 全局配置和全局状态。
3. 为了让功能“自动可用”，在模块导入时就初始化对象。
4. 为了支持更多场景，再继续在已有分支上加参数和判断。
5. 最终任何改动都会影响测试、文档、命令流、存储和网络层。

PromptPro 现在基本就在这个阶段。

如果继续沿着这个方向加功能，后面会越来越明显地出现这些问题：

- 改一个命令，别的命令跟着出问题
- 修一个 provider，测试环境又坏掉
- 文档和实际行为越来越不一致
- 新功能只能“能跑”，很难做到“稳定可维护”

所以这次整改不应该理解为“代码整理一下”，而应该理解为：

**重新给项目建立边界和秩序。**

## 设计原则

后续优化时，建议所有改动都遵守下面几条原则，否则很容易重新滑回现在的状态。

### 1. 导入必须无副作用

任何模块被 import 时，都不应该：

- 创建目录
- 写配置文件
- 访问网络
- 创建全局运行态对象

导入只做定义，不做执行。

### 2. CLI 不是业务层

CLI 的职责应该只有：

- 收集输入
- 调用 service
- 渲染输出
- 处理退出码

凡是“怎么优化 prompt”“怎么决定是否提问”“是否保存历史”这类逻辑，都不该埋在 CLI。

### 3. 所有外部依赖都要可替换

这里的外部依赖包括：

- 文件系统
- 网络请求
- 剪贴板
- 时间
- 当前路径 / home 目录

如果这些依赖不能替换，测试就会一直脆弱，功能也很难组合。

### 4. 默认路径必须短

用户最常见的路径应该是：

- 输入一句 prompt
- 选择一个简单模式，或完全不选
- 得到 1 到 3 个稳定结果

不要把用户引导到“先理解 7 种框架、多个命令、历史、文档系统、provider 配置”。

### 5. 宣传语必须小于等于真实能力

README 不是产品理想，而是能力说明书。

如果实现只是：

- 基于规则做模式归类
- 调 LLM 做 prompt 重写

那就应该这么写，不要包装成“智能理解引擎”“核心创新系统”。

```text

  app/
    services/
    models/
  infra/
    llm/
    storage/

    commands/
```

### 8. 加一层应用编排，不要让功能彼此直连

问题：

- 现在很多模块之间是直接互相调用
- 这会导致“一个小功能需要知道另一个模块的内部细节”

建议：

- `PromptOptimizationService` 负责主流程
- `FrameworkRecommendationService` 负责模式推荐
- `ClarificationService` 负责补充提问
- `HistoryRepository` 负责保存历史
- `LLMClient` 只负责和 provider 通信

这样每一层都只知道自己该知道的事。

## 风险判断

### 最大风险不是技术，而是继续发散

如果不先做产品收缩，而是一边重构一边继续加功能，最后很可能两头都失败：

- 结构没整理干净
- 用户路径更复杂
- 文档继续超卖

### 第二个风险是“半重构”

最危险的状态不是不重构，而是只拆了一半：

- 文件变多了
- 逻辑没有真正解耦
- 全局状态还在
- 测试还是脆弱

这种状态维护成本会比现在还高。

### 第三个风险是过度设计

这个项目还没到需要复杂架构图和大量抽象层的程度。

所以重构目标不是“高级架构”，而是：

- 去副作用
- 去耦合
- 去伪能力宣传
- 保证默认路径好用

## 最终目标

把这个项目从“功能很多但不稳的 prompt 工具箱”，收敛成：

**一个边界清晰、可测试、可维护、输出稳定的 Prompt 重写 CLI。**

进一步说，理想状态应该是：

- 用户第一次用，不需要理解内部设计就能得到结果
- 开发者改一处逻辑，不会牵连整条链路
- 测试可以在干净环境稳定运行
- 文档说的每一句话，都能在代码里找到对应能力
